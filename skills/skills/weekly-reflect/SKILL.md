# Weekly Reflect — 주간 횡단 회고

Step 7 (Reflect)을 자동화하는 스킬. 작업 단위 회고가 아닌 **7일치 횡단 회고**.

## 호출 트리거

| 트리거 | 행동 |
|---|---|
| `/회고` 또는 "주간 회고해줘" | 즉시 실행 |
| "이번 주 어땠어 회고하자" 등 자연어 | 즉시 실행 |
| 일요일 밤 cron (추후 schedule 스킬로 자동화) | 자동 실행 |
| 큰 마일스톤 종료 후 사용자 요청 | 임시 실행 |

## 실행 단계

### Step 1: 기간 확정
- 기본: 직전 일요일 ~ 토요일 (또는 최근 7일)
- 사용자가 명시하면 그 기간
- 타임존: KST

### Step 2: 데이터 수집 (병렬)

#### 2-1. 노션 로그 DB
```
mcp__notion__API-query-data-source 사용
data_source_id: 4c86dd0a-4d34-41b6-a19f-c6b5c037258f의 data_source_id
필터: 일시 BETWEEN [기간 시작] AND [기간 끝]
정렬: 일시 ASC
```
DB ID(4c86dd0a...)와 data_source_id는 다름. 먼저 `API-retrieve-a-database`로 data_source_id 확인.

#### 2-2. 메모리 갱신 분석
- `~/.claude/projects/.../memory/MEMORY.md` 변경 이력 (있으면 git log, 없으면 mtime)
- 새로 추가된 메모리 파일 목록

#### 2-3. 프로젝트별 작업 흔적
- 6개 프로젝트 폴더의 mtime 7일 이내 파일 목록
- 각 프로젝트의 `tasks/todo.md` 있으면 완료/미완료 추출

### Step 3: 분석 (Claude)

수집된 데이터를 Claude가 직접 분석. 다음 5개 섹션을 채움:

#### 3-1. 잘 풀린 패턴 3개 + 그 이유
- 빠르게 결론까지 간 작업
- 한 번에 성공한 패턴
- 사용자 만족 신호 (피드백 톤, "ㅇㅇ" 빈도)

#### 3-2. 막힌 지점 + 근본 원인
- 여러 번 재시도한 작업
- "왜 안 됐지" 질문이 반복된 영역
- 미완료 todo

#### 3-3. 다음 주 우선순위 후보
- 미완료 todo 중 영향 큰 것
- 사용자가 "나중에" 한 항목
- 이번 주 발견된 새 기회

#### 3-4. 새 규칙 후보
- 같은 피드백이 2번 이상 반복 → CLAUDE.md / rules / auto memory 후보
- 어디 저장할지 제안 (글로벌 vs. 프로젝트 vs. 메모리)

#### 3-5. 메모리 갱신 제안
- 추가/수정/삭제할 auto memory 항목
- 사실 변화 (예: 프로젝트 상태, 사용자 역할 변화)

### Step 4: 회고 DB에 행 추가

#### DB 정보 (등록 완료, 2026-05-03)
- **DB 이름**: 주간 회고
- **DB ID**: `35553d16-da25-8016-bbb0-de7e0a2ddc7d`
- **DB URL**: https://www.notion.so/35553d16da258016bbb0de7e0a2ddc7d
- **부모 페이지**: "Claude 회고" (`35553d16-da25-8041-a142-d644924a895d`)
- **통합 권한**: `mcp__notion__` (Claude Code) ✅ 확인됨

#### 페이지 생성 도구
`mcp__notion__API-post-page` 사용:

```
parent: { "type": "database_id", "database_id": "35553d16-da25-8016-bbb0-de7e0a2ddc7d" }
```

#### 속성 매핑 (정확한 이름·타입 준수)

| 속성 이름 | 타입 | 값 |
|---|---|---|
| `이름` | title | `주간 회고 YYYY-MM-DD ~ YYYY-MM-DD` |
| `시작일` | date | 기간 시작 (ISO 8601) |
| `종료일` | date | 기간 끝 (ISO 8601) |
| `주차` | rich_text | `2026-W18` 형식 |
| `잘된 패턴` | multi_select | 옵션: `빠른클로저` / `4-way통과` / `재사용성` / `Think적중` |
| `막힌 지점` | multi_select | 옵션: `재시도반복` / `요건모호` / `스킬부재` / `Think누락` |
| `다음주 우선순위` | rich_text | 핵심 1~3개 한 줄로 |
| `새 규칙 후보` | number | 발굴 규칙 후보 개수 |
| `메모리 갱신` | number | 메모리 추가/수정 항목 개수 |

> ⚠️ 새 옵션 필요시 multi_select에 추가 가능 (DB가 자동 학습). 단, 일관성 위해 기존 옵션 우선 매칭.

#### 본문 (children 마크다운)
5개 섹션을 본문에 넣음:
1. 잘 풀린 패턴 3개 + 그 이유
2. 막힌 지점 + 근본 원인
3. 다음 주 우선순위 후보
4. 새 규칙 후보 (저장 위치 제안)
5. 메모리 갱신 제안

### Step 5: 사용자 검토 보고
회고 페이지 URL과 함께 다음 보고:
- 핵심 인사이트 3줄 요약
- 사용자 승인 필요 항목 (규칙 추가, 메모리 갱신 등)
- 승인되면 즉시 반영 (Step 6의 "즉시 반영 원칙" 적용)
