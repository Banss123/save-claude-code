# 밸류체인 자동화 프로젝트

> 요식업 배달 컨설팅 업체의 온보딩 업무 자동화/효율화
> 작성일: 2026-04-09

---

## 1. 프로젝트 개요

### 사업 모델
- 요식업 자영업자 대상 배달 플랫폼 컨설팅 업체
- 배달 4사: 배민, 쿠팡이츠, 요기요, 땡겨요

### 핵심 문제
- 실무 과부하로 신규 고객 확장 불가
- 솔루션 담당자 1명이 전체 온보딩 처리 중

### 목표
- 가능한 최대한의 자동화/효율화
- **정확성이 최우선** — 실수 없는 산출물 + 강력한 검수 프로세스

### 사용 중인 도구
- 엑셀, 구글시트, 슬랙
- Python 로그인 자동화 스크립트 (배민 사장님사이트)
- Claude 프롬프트 기반 반자동 문서 생성

---

## 2. 밸류체인 구조 및 우선순위

| 단계 | 핵심 업무 | 우선순위 |
|------|-----------|----------|
| 1. 계약(영업) | 신규 고객 확보 | - |
| 2. 온보딩 | 솔루션계획서, 메뉴판 가안, CPC/CMG 광고값 제안 | **1순위** |
| 3. 플랫폼 적용 | 솔루션/메뉴판을 배달 플랫폼에 실제 세팅 | 1순위 (체크리스트 효율화) |
| 4. 매출 모니터링 | 배민/쿠팡이츠/요기요/땡겨요 4사 추이 관리 | 2순위 |
| 5. 정산/세무 | 정산금 처리, 손익계산서 제공+분석 | 3순위 |

> 플랫폼 적용은 API 없음 + 실수 불가 → 완전 자동화 불가, 체크리스트/가이드 자동 생성으로 효율화

## 3. 현재 시스템 분석 (반자동 프로세스)

### 3-1. 메뉴판 가안 프로세스

```
수동: 배민 사장님사이트 접속 → 3개 메모장에 복사
  ├─ [가게명]_가격표.txt    (메뉴관리 → 메뉴 목록 전체 복사)
  │   → 메뉴그룹명, 메뉴명, 가격(배달/포장), 구성, 설명
  ├─ [가게명]_옵션리스트.txt (메뉴관리 → 각 메뉴 클릭 → 옵션 영역 복사)
  │   → 옵션그룹명, 조건(필수/선택), 옵션 항목, 추가가격
  └─ [가게명]_가안.txt      (솔루션 담당자 직접 작성)
      → 메뉴그룹/메뉴/옵션 변경 제안 내용

자동: Claude 프롬프트 + txt 3개 + (선택) 예시 xlsx
  → 현안 vs 가안 비교 XLSX 생성
  → 시트1 "현안" / 시트2 "가안"
  → C열부터 시작 (A-B 여백), 열 순서:
     메뉴그룹명|메뉴명(가격/배달)|구성(설명)|옵션명|옵션그룹명(조건)|옵션목록(속성/추가격)|가격가

수동: 검수 체크리스트로 QA
```

#### 메뉴판 가안 검수 체크리스트
- [ ] 현안 시트의 메뉴 수 = 가격메뉴.txt의 메뉴 수와 일치
- [ ] 가안 시트의 메뉴 수 = 현안과 동일 (메뉴 추가/삭제가 없는 한)
- [ ] 옵션그룹 번호가 올바른 메뉴에 할당되어 있는지
- [ ] 가안에서 변경된 셀만 볼드 처리되어 있는지
- [ ] 변경 없는 셀은 현안과 동일한 내용인지
- [ ] 가격이 숫자로 입력되어 있는지 (문자열 아님)
- [ ] 사이즈별 분리 메뉴가 올바르게 병 분리되어 있는지

### 3-2. 솔루션 계획서 프로세스

```
수동: 솔루션방향.txt 직접 작성 (솔루션 담당자)
  → 가게 정보 (가게명, 작성일, 특보일)
  → 배민 CPC 단가(목표), 배민 울 광고비(예측), 배민 CMG(비용)
  → 현재 객단가 & 최소주문금액
  → (필요시) 추가 KPI: 현재 CTR / 현재 CVR / 지역 평균 CVR 등
  → 현안 요약 (메뉴판, 광고, 운영)
  → 가안 요약 (메뉴판, 광고, 운영)
  → 핵심 한줄 메시지

자동: Claude 프롬프트 + txt + (선택) 예시 docx
  → 솔루션 계획서 DOCX 생성

수동: 검수 체크리스트로 QA
```

#### 솔루션 계획서 DOCX 구조
- **1페이지: 표지 + 요약표**
  - 타이틀, 가게정보/담당자, 기간/계약
  - 핵심 KPI 4건 이상 (배민 CPC, 배민 울 광고비, 쿠팡 CMG, 객단가 등)
  - 비교표: "지금"(현안) vs "앞으로"(가안)
  - 핵심 메시지 강조 박스

- **2페이지~: 세부 계획** (섹션 구분)
  - a. 배민 기본 세팅 (가격상표 화폐, 사장님 공지, 목시할인, 메뉴명 키워드, 팁팝업 등)
  - b. 광고 전략: 배민 (CPC 단가/예산/위치기간)
  - c. 광고 전략: 쿠팡이츠 (CMG 설정, 비율/조건)
  - d. 대표메뉴 시장 (6건 채우기 여부)
  - e. 수수료 구조 (성과 기반 수수료)
  - f. 운영 원칙 (금지사항, 임시울자/운영시간 변경 금지 등)

#### 솔루션 계획서 디자인 규칙
- 색상: 섹션 구분선 #F2F2F2, 비교표 다크 #1E1E1E/#202020, 핵심 메시지 #2B7A4B
- 폰트: Malgun Gothic 10pt, 타이틀 Bold 24-28pt, 섹션 Bold 12-14pt
- 레이아웃: A4, 여백 약 2cm

#### 솔루션 계획서 검수 체크리스트
- [ ] 1페이지에 KPI 4건 박스가 정확히 표시되는지
- [ ] 비교표의 "지금/앞으로" 내용이 txt 원본과 일치하는지
- [ ] 핵심 메시지가 박스 안에 들어가 있는지
- [ ] 섹션 구분자(①~⑤)가 모두 포함되어 있는지
- [ ] 항목 번호가 1부터 연속으로 끝까지 이어지는지
- [ ] 수수료 구조 테이블이 정확한지
- [ ] "목표 매출 미달성 시 수수료 미발생" 강조 여부
- [ ] 운영 원칙 경고 문구 볼드 처리 여부
- [ ] 폰트랑 통일 여부
- [ ] txt에 없는 내용이 임의로 추가되지 않았는지

#### 솔루션방향 txt 작성 팁 (필수/선택 항목)
**필수 포함 항목:**
1. 기본 KPI (CPC/예산/CMG/객단가 최소 3개)
2. 현안-가안 요약 (메뉴판/광고/운영 각 3개)
3. 배민 세팅 세부사항 (최소 5개 항목)
4. 광고 전략 (배민 CPC + 쿠팡 CMG)
5. 수수료 구조 (매출 구간별 요율)
6. 운영 원칙 (금지 사항)

**선택 포함 항목:**
- 한그릇 입점 구성
- 메뉴모음컷 촬영 계획
- 리뷰 반영 사항
- 쿠팡이츠 쿠폰/즉시할인 전략
- 가게배달 + 배달팁 설정

## 5. 자동화 목표 아키텍처 (v3 — Claude API + 슬랙 봇)

> v1: Claude 채팅 기반 (반자동)
> v2: Claude API + 구조화 출력 (코드 파이프라인)
> **v3: v2 + 슬랙 봇 인터페이스 (비개발자 실행 가능, 최대 자동화)**
> AI 역할을 "판단"에만 한정하고, 문서 포맷팅은 결정적 코드로 분리
> 인터페이스: 슬랙 슬래시 커맨드 → 파이프라인 트리거 → 슬랙으로 결과 배달

### 현재 방식 vs 개선 방식 비교

| 비교 항목 | 현재 (채팅 기반) | 개선 (API 기반) |
|-----------|-----------------|----------------|
| AI 역할 | 판단 + 포맷팅 혼합 | **판단만** (구조화 출력) |
| 문서 포맷 | 매번 다름 | **항상 동일** (코드 고정) |
| 오류 추적 | 어디서 틀렸는지 모름 | JSON 단계 vs 문서 단계 분리 |
| 검수 | 수동 체크리스트 | **자동 검증 가능** |
| 대량 처리 | 1건씩 수동 | **Batch API로 동시 처리 (50% 할인)** |
| 비용 | Opus 매번 풀 요청 | **프롬프트 캐싱 (90% 절감)** |

### 파이프라인 흐름 (v3)

```
[슬랙] /온보딩 홍콩반점
    │
    ▼
⓪ 슬랙 봇 수신 ─────── Bolt for Python → 요청 파싱 + 진행 스레드 생성
    │
    ▼
① 데이터 자동 수집 ─── Playwright로 배민 사장님사이트 자동 로그인 + 스크래핑
    │                    (기존 Python 스크립트 확장)
    │                    → 슬랙 진행 상태 업데이트
    ▼
② Claude API 분석 ──── Structured Output (구조화 출력)
    │                    → 입력 데이터 분석 → 솔루션 JSON 산출
    │                    → AI는 "판단"만, 포맷팅은 안 함
    │                    → Pydantic 스키마로 출력 형식 강제
    ▼
③ JSON 자동 검증 ───── 스키마 검증 + 비즈니스 룰 검증
    │                    → 이상 발견 시 즉시 플래그 + 슬랙 경고
    ▼
④ 결정적 코드로 문서 생성 ── openpyxl → XLSX, python-docx → DOCX
    │                         → 검증된 JSON → 항상 동일한 문서 출력
    │                         → 디자인 규칙(폰트,색상,레이아웃) 코드 고정
    ▼
⑤ 문서 자동 검수 ──── 입력 JSON ↔ 출력 문서 교차 검증
    │                   → 체크리스트 자동 적용
    ▼
⑥ 슬랙 배달 ────────── 파일 첨부 + 검수 리포트 → 슬랙 스레드
    │                    → 정상: 문서 첨부 + "검수 통과"
    │                    → 이상: 문서 + 경고 + 상세 리포트
    ▼
[완료]

배치 모드: /온보딩-배치 업장1, 업장2, 업장3
    → Batch API 동시 처리 (50% 할인) → 완료 시 일괄 보고
```

### 활용하는 Claude API 기능

| 기능 | 역할 |
|------|------|
| **Structured Output** | 솔루션 JSON을 Pydantic 스키마로 강제 → 누락/오류 원천 차단 |
| **Adaptive Thinking** | 솔루션 판단 시 Claude가 사고 깊이 자동 조절 → 정확도 향상 |
| **Prompt Caching** | 솔루션 규칙/시스템 프롬프트 캐싱 → 반복 요청 시 90% 비용 절감 |
| **Batch API** | 여러 업장 동시 처리 → 50% 비용 할인 |
| **Code Execution** | (대안 경로) 서버 샌드박스에서 직접 문서 생성 가능 |

## 6. 실행 계획 (v2)

### Phase 1: 데이터 수집 자동화
1. 프로젝트 구조 세팅
2. 기존 Python 로그인 스크립트 확인 → Playwright 기반 메뉴/옵션 데이터 자동 추출
3. 추출 데이터 → 표준 JSON 포맷 자동 변환

### Phase 2: Claude API 파이프라인 구축
4. 솔루션 JSON 스키마 정의 (Pydantic)
5. Claude API + Structured Output으로 솔루션 자동 생성
6. JSON → XLSX 변환 코드 (openpyxl)
7. JSON → DOCX 변환 코드 (python-docx)
8. 프롬프트 캐싱 적용

### Phase 3: 자동 검수 시스템
9. JSON 스키마 검증 (입력 유효성)
10. 비즈니스 룰 검증 (메뉴 수 일치, KPI 존재 등)
11. 입력 JSON ↔ 출력 문서 교차 검증
12. 이상 항목 리포트 자동 생성

### Phase 4: 솔루션 규칙 엔진 (규칙 확정 후)
13. 솔루션 판단 규칙 체계화
14. 과거 4-5건 패턴 학습
15. 가안/솔루션방향 자동 초안 생성

## 7. 기술 스택 (v3)

| 영역 | 도구 | 사유 |
|------|------|------|
| **인터페이스** | **Slack Bot (Bolt for Python)** | 기존 슬랙 워크플로우 활용, 학습 비용 제로, 파일 배달+알림 자연스러움 |
| 브라우저 자동화 | Playwright (Python) | Selenium보다 빠르고 안정적, 비동기 지원 |
| 데이터 처리 | Python + Pydantic | 타입 안전 JSON 스키마 정의 + 검증 |
| AI 판단 | Claude API (Structured Output) | 구조화된 JSON으로 솔루션 산출 |
| XLSX 생성 | openpyxl | 결정적 엑셀 생성 (한글 폰트 정밀 제어 최강) |
| DOCX 생성 | python-docx | 결정적 워드 생성 (한글 폰트 정밀 제어 최강) |
| 비용 최적화 | Prompt Caching + Batch API | 캐싱 90% 절감 + 배치 50% 할인 |
| 데이터 관리 | Google Sheets (MCP) | 기존 워크플로우와 호환 |
| 미래 전환 | Anthropic Managed Agents | Beta 졸업 후 오케스트레이터 전환 검토 |

### 기술 스택 선정 근거 (2026-04-09 비교 분석)

- **Python 선택**: openpyxl/python-docx 한글 포맷팅 최강 + 기존 로그인 스크립트 호환 + Pydantic↔Structured Output 직접 연동
- **TS/Go 탈락**: 문서 라이브러리 성숙도 부족, 기존 스크립트 재작성 필요
- **노코드(n8n, Make) 탈락**: XLSX/DOCX 정밀 포맷팅 불가, Structured Output 직접 지원 제한

- **Computer Use 탈락**: Playwright 대비 느리고 비용 높고 정확성 낮음 (이 프로젝트는 정확성 최우선)

## 8. 운영 환경별 가능 범위

### Claude Code (개발 도구) — 시스템 구축용
- 이 자동화 시스템을 **만드는** 도구
- 시스템 완성 후에는 불필요

### 일반 Claude AI (claude.ai) — 현재 반자동 방식
- txt 붙여넣기 + 프롬프트 → XLSX/DOCX 생성 (현재 하고 있는 것)
- 수동, 1건씩, 검수도 수동
- 자동화/배치/자동검수 불가

### 슬랙 봇 + Python 앱 (최종 산출물) — 실제 운영용
- 슬랙에서 `/온보딩 [업장명]` → 자동 스크래핑 → Claude API → 문서 생성 → 자동 검수 → 슬랙으로 결과 배달
- 비개발자도 슬랙에서 바로 실행
- Claude Code 없이 독립 실행
- 배치 모드: `/온보딩-배치` → 다건 동시 처리
