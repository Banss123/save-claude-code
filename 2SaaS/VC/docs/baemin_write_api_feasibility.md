## A. 수정 API 존재 가능성

| 소스 | 가능성 | 근거 |
|------|:---:|------|
| 배민 공식 Open API (Write) | **낮음** | `OrderRelaySampleCode`(주문 중계)만 제휴사 공개. 메뉴·광고 Write는 `self.baemin.com` UI 또는 파트너전담센터(1600-2111) 오프라인 접수뿐. 자회사 프랜디는 본사용 내부 도구라 외부 SDK 없음. |
| 내부 REST/GraphQL (HAR 역공학) | **높음** | `self.baemin.com`은 SPA로 `biz-member.baemin.com` 세션 쿠키 기반 XHR 호출이 일상. 기존 수집 스크레이퍼(`src/scraper/baemin.py`)가 해당 세션을 이미 유지 → 동일 세션 재사용으로 Write 호출 가능성 매우 높음. 경로·스키마는 HAR 캡처로 확정. |
| Playwright DOM 조작 | **중** | 폴백용. `probe_results/` 확인 결과 클래스가 `menuItem-module__menu--R6db` 등 **CSS Modules 해시** → 빌드마다 변화로 셀렉터 취약. UI 텍스트 앵커(`메뉴판 편집`)가 그나마 안정. API 실패 시 보조만. |

**결론**: 담당자 HAR 캡처 → `httpx`로 XHR 재호출이 최적. Playwright는 storage_state·폴백·증빙 스크린샷 용도.

## C. 안전장치 (3중 + 확장)

1. `--dry-run`: `session.send()` 직전 차단, 변경 예정 리포트만 출력
2. 스냅샷: Write 직전 GET → `snapshots/{매장}/{타임스탬프}.json`, diff를 슬랙에 동봉
3. 슬랙 승인 게이트: 카테고리 분할(메뉴/옵션 2단계), 승인 토큰 5분 TTL
4. Rate limit 가드: ≤ 10건/분 셀프 스로틀(공식 제한값 미공개, 보수 적용)
5. 롤백 드라이런: 스냅샷 재적용 스크립트가 dry-run 통과해야 Write 허용
6. 변경이력 교차검증: 배민 `변경이력` 탭 조회로 실반영 확인

| 항목 | 등급 | 근거 |

| 실계정 영향 | **고** | 가격 오기입 = 즉시 매출 타격. **테스트 계정 필수**(담당자 명의 테스트 매장 권장). |

## E. 구현 로드맵

**Phase 1 — 엔드포인트 매핑** (담당자 1회 30분)
- 테스트 매장 로그인 → DevTools Network 녹화 → 메뉴·옵션 1건씩 "원래 값 그대로" 저장 → HAR export → `src/writer/endpoints.py` 고정

**Phase 2 — Dry-run PoC** (1매장, 1~2일)
- `src/writer/baemin_writer.py` (httpx + storage_state) 스냅샷+dry-run만 구현. Write는 주석 처리. 스냅샷이 UI와 100% 일치 검증.

**Phase 3 — 슬랙 승인 + 실 Write** (1주)
- 메뉴 1건 Write → `변경이력` 확인 → 카테고리 전체 → 옵션 확대. 롤백 시나리오 1회 실행.

## F. 권장 MVP 범위

**메뉴·옵션 텍스트(이름·설명·가격)만** — 기존 확정안 유지.

근거:
- 담당자 주업무의 80%+ 커버 (메뉴 가안이 핵심 산출물)
- 가격 오기입 리스크는 스냅샷 롤백 범위 내
- 광고는 배민 "스마트 모드"가 대체 → 자체 구현 ROI 낮음
- 운영정보는 월 1회 이하 변경 → 수동 잔류 충분

광고·운영정보는 MVP 안정화 후 분기 단위 확장.
