> 배민 상세: `docs/baemin_write_api_feasibility.md`. 본문은 나머지 3사 + 4사 매트릭스 + MVP 순서. 공개 근거 없는 항목은 "확인 불가"로 명시.

## 2. 사별 상세

### 2.1 배민 (요약)

내부 XHR 역공학 + 6중 방어로 Phase 3 실 Write. 메뉴·옵션 텍스트 MVP. 상세 `baemin_write_api_feasibility.md` §E.

### 2.2 쿠팡이츠

**경로**: `store.coupangeats.com` (SPA). 공식 파트너 Open API는 쿠팡 본사 e-commerce(WING)용. 쿠팡**이츠** 전용 Merchant API 공개 문서 없음.

**셀프서비스 (2024~2025)**: 메뉴·가격·옵션그룹·이미지 셀프 수정 순차 오픈. 단 **변경 후 시스템 검토 1~2일 지연** — 배민 즉시 반영과 다른 비동기 승인. 자동화도 "Write 큐잉 → 최대 2일 반영"으로 설계 필요.

**약관**: 쿠팡 공통 약관에 "자동화 검색·요청, 자동화 에이전트·스크립트" 금지 명시. 사장님 포털 전용 약관 비공개이나 동일 정책 추정. 본사 쿠팡에서 "어뷰징 계정 전면 정지" 이력 — 탐지 감도 4사 중 최상 추정.

**역공학**: 중. HAR 캡처 방식 동일하나 **약관 명시 금지로 법적 리스크 배민보다 높음**.

### 2.3 요기요

**경로**: `partner.yogiyo.co.kr`, `ceo.yogiyo.co.kr`. 공식 Open API 미확인.

**셀프서비스**: 메뉴·옵션·가격·이미지 직접 수정. 카카오톡 "요기요사장님" 채널로 수정 요청 가능 — **고객센터 채팅봇이 Write 경로로 정식 존재**. 이 채널 API화 여부 탐색 가치 있음 (담당자 확인 필요).

**약관·탐지**: 자동화 금지 명시 조항 미확인. 과거 쿠폰 번호 매크로 공격 이력으로 봇 탐지 인프라 보유 추정.

**역공학**: 중. 배민과 유사 세션 쿠키 기반 SPA. HAR 패턴 동일 적용 가능.

### 2.4 땡겨요

**경로**: `boss.ddangyo.com` (신한은행 운영).

**현황**: 공개 API 문서·개발자 포털 전무. 신한 Open API는 금융 전용, 땡겨요 무관. 메뉴 이미지 최초 설정은 **"땡겨요 제공 → 해피콜로 사장님 이미지 반영"** — 사람 매개 워크플로우 일부 포함.

**MVP 제외 근거**: 시장 점유율 최하위, 대상 매장 미입점 다수, 공개 자산 전무. 배민 안정화 후 재평가.

## 3. 6중 안전장치 4사 적용 가능성

| 안전장치 | 배민 | 쿠팡이츠 | 요기요 | 땡겨요 |
|---|:---:|:---:|:---:|:---:|
| dry-run | O | O | O | 적용 대상 아님 |
| 스냅샷 | O | O (승인 지연 고려) | O | - |
| 슬랙 승인 | O | O | O | - |
| Rate limit 가드 | O | O (보수적, ≤5/분 권장) | O | - |
| 롤백 | O | **△** (1~2일 지연으로 롤백 타이밍 복잡) | O | - |
| 변경이력 교차검증 | O | O (포털 이력 탭) | O (미확인) | - |

쿠팡이츠는 승인 지연으로 "Write → 대기 → 검증 → 롤백" T+1~2 루프 필요. 배민 스냅샷 로직 이식 불가 — **별도 큐 모델** 설계.

## 4. 권장 MVP 순서

**배민 → 쿠팡이츠 → 요기요 → 땡겨요**

1. **배민**: 약관 자동화 금지 **미확인**, 기존 스크레이퍼 재활용, 즉시 반영 → 검증 사이클 최단.
2. **쿠팡이츠**: 기술 가능하나 약관 명시 금지 + 승인 지연 + 탐지 감도 최상. 배민 안정화 후 "셀프서비스 범위 내 자동화" 명분으로 보수 진입.
3. **요기요**: 시장 규모 작고 약관 불명. 카카오톡 Write 채널 API화 여부가 관건.
4. **땡겨요**: 공개 자산 전무 + 해피콜 의존 → MVP 제외, 후기 재평가.

## 5. 법적 리스크 (한국)

- **대법 2021도1533 (2022.5.12)**: 경쟁사 API 크롤러로 DB 복제 → 정보통신망 침입·DB권·업무방해 성립. 쟁점은 "경쟁사 데이터 대량 수집" → **자기 매장 자기 계정 Write 직접 적용 어려움**. 단 "약관 위반 + 자동화" 결합 시 업무방해 성립 가능성 잔존.
- **실무 결론**: 판례 리스크 낮음. 쿠팡이츠는 약관 명시 금지로 **계정 정지 = 사업 중단** 리스크가 더 큼.

## 6. 실사례·참고

- **오늘얼마**: 배민·요기요·쿠팡이츠 3사 ID/PW 연동. **땡겨요 미지원**. Read 전용. 업계 선행주자도 Write 미진입.
- **포스피드·토스포스·오케이포스**: 주문 Read 통합만. 메뉴 Write 자동화 공개 사례 없음.
- **결론**: 4사 Write 자동화 공개 선행 사례 전무 → 초기 **저속 운영 + 증거 보전 + 담당자 승인 로그**로 선례 구축.
