## 이슈 1. Playwright 효율화 (10분 → 2~3분 목표)

### 현황
- 현재 Playwright로 DOM 스크래핑 방식, 매장당 약 10분 소요
- 페이지 로드 + 렌더링 대기 시간이 대부분

### 권장 접근법
**DOM 스크래핑 → 내부 API JSON 가로채기 전환**

1. **1회성 셋업 (플랫폼별 1회)**
   - Chrome DevTools → Network 탭 녹화로 사장님사이트 내부 API 엔드포인트 + 인증 헤더 + 응답 JSON 구조 캡처
   - 예: 배민은 `/api/v1/shops/{id}/menus`, `/api/v1/ads/campaigns` 등 내부 JSON API로 데이터 송수신
   - 매핑을 `src/scraper/endpoints.py`에 고정 (경로 + 필요 헤더 + Pydantic 응답 스키마)

2. **운영 (매번)**
   - `storage_state`로 쿠키/세션 저장 → 재접속 시 로그인 단계 스킵 (만료 시 재로그인)
   - 리소스 차단(이미지/폰트/광고) → 로드 시간 30~50% 단축
   - 매장별 `asyncio.gather` 병렬화 (단, 동일 계정 동시 로그인 제약 확인 필요)

### 리스크 / 주의
- 플랫폼이 API 구조 변경 시 재매핑 필요 (DOM보다는 훨씬 덜 빈번)
- 배민 외 쿠팡이츠·요기요·땡겨요도 플랫폼별 1회씩 셋업 필요 (API 구조 상이)
- 2FA/캡차: 배민은 관대하나 쿠팡이츠는 걸릴 수 있음 → 최초 수동 인증 + storage_state 저장으로 우회

### 미팅 결정사항

- [ ] 2FA 대응 방식 (SMS 중계 vs 초회 수동)

### 담당자 PC 실행 체크리스트

**미팅 전 (유일 블로커)**

## 이슈 2. 메뉴판 xlsx 템플릿 정형화

### 현황
- 담당자 결과물이 프로젝트 표준과 다른 템플릿으로 생성됨
- 원인 추정: 기존 코드가 과거 사례 xlsx를 직접 참조하거나, 템플릿 상수가 하드코딩 안 되어 있어 데이터별로 다른 결과 생성

### 권장 접근법
**단일 진입점 `xlsx_builder.py` + 고정된 `XLSX_CONFIG` 사용**

CLAUDE.md에 이미 정의된 표준:
```python
XLSX_CONFIG = {
    "start_col": "C",
    "columns": ["메뉴그룹명", "메뉴명(가격/배달)", "구성(설명)",
                "옵션명", "옵션그룹명(조건)", "옵션목록(속성/추가격)", "가격가"],
    "sheets": ["현안", "가안"],
    "bold_changed_cells": True,
    "font": "맑은 고딕",
    "font_size": 10,
}
```

- 어떤 매장이든 동일 템플릿 보장 (결정적 생성)
- 변경 셀만 볼드 처리 → 가독성 확보
- 담당자 측 기존 코드를 이 빌더로 치환

### 리스크 / 주의
- 과거 담당자 결과물과 포맷이 달라지면 레퍼런스 매장들의 재생성 필요할 수 있음
- 가독성 세부 튜닝(열 너비, 병합, 구분선)은 실제 샘플 보면서 조정해야 함

### 미팅 결정사항
- [ ] 표준 템플릿으로 통일 여부 확정
- [ ] 가독성 추가 요구사항 수집 (담당자 선호 포맷)
- [ ] 기존 생성물 재생성 여부 (과거 매장 소급 적용?)

### 담당자 PC 실행 체크리스트
- [ ] 담솔 지웰시티점 `가안.json` 1건으로 표준 템플릿 샘플 xlsx 생성 → 비교
- [ ] 가독성 피드백 반영 (열 너비, 구분선 등)
- [ ] `xlsx_builder.py` 단일 진입점으로 기존 코드 치환
- [ ] 기존 생성물 vs 신규 생성물 diff 확인

### 즉시 준비 가능
- 담솔 지웰시티점 `가안.json` 경로 공유받으면 **지금 샘플 xlsx 생성 가능** → 미팅에 실물 지참 가능

## 이슈 3. 가안 → 플랫폼 역적용 (상세 설정 자동 수정)

### 현황
- Playwright로 정보를 **긁어오는** 것은 되는 상태
- 뽑힌 가안을 실제 플랫폼들에 **역으로 반영**(메뉴 수정, 옵션 변경, 광고 설정 등)하는 확장이 담당자 니즈

### 권장 접근법
**기술적으로 가능하지만, 3중 안전장치 필수**

1. **`--dry-run` 모드**
   - 실제 API 호출 없이 "무엇이 변경될지"만 리포트
   - 슬랙에 "변경 예정 12건" 상세 출력

2. **변경 전 스냅샷 저장**
   - 수정 직전 현재 상태를 JSON으로 덤프 → `snapshots/{매장}_{타임스탬프}.json`
   - 문제 발생 시 롤백용

3. **슬랙 승인 게이트**
   - "아래 12건 변경 적용할까요? ✅ / ❌"
   - 명시 승인 없이는 실행 안 됨
   - 일괄 적용 대신 배치별(메뉴 / 옵션 / 광고) 분할 승인 옵션

### 리스크 / 주의
- **실제 운영 매장에 영향 가는 작업** → CLAUDE.md 우선순위 체인 "정확성 > 안전성 > 효율성 > 속도" 최대 준수
- 플랫폼이 변경 API를 제공하지 않으면 DOM 조작 방식 불가피 (더 불안정)
- 4사 API 구조 상이 → 플랫폼별 개별 구현 필요
- 롤백 시나리오 사전 검증 필수

### 미팅 결정사항
- [ ] MVP 범위: 배민만 먼저? 4사 동시?
- [ ] 역적용 범위: 메뉴/옵션만? 광고 설정까지?
- [ ] 승인 게이트 단위: 매장 전체 / 변경 카테고리별 / 개별 항목별
- [ ] 롤백 책임: 자동 롤백 시도 vs 수동 개입

### 담당자 PC 실행 체크리스트
- [ ] **테스트 매장 선정·확보** (실영업 영향 없는 계정, 담당자 본인 테스트 계정 권장)
- [ ] 이론 합의 후, 테스트 매장에서 PoC (메뉴 1건 수정)

## 이슈 4. 케이스별 로직 엔진 (조건부 솔루션 추천)

### 현황
- 담당자 니즈: 매장 상태별 케이스에 따라 솔루션 자동 추가
- 예: 리뷰 이벤트 미설정 매장 → 리뷰 이벤트로 초기 리뷰 축적 제안

### 권장 접근법
**룰 기반(YAML) + Claude API 문장화 조합**

```yaml
# rules/suggestions.yaml (예시)
- id: review-event-missing
  when: shop.review_event.enabled == false
  then:
    suggestion: "리뷰 이벤트 설정으로 초기 리뷰 축적"
    section: 배민 기본 세팅
    priority: high

- id: cpc-soji-low
  when: ads.cpc_consume_rate < 0.3
  then:
    suggestion: "가게대표·콘텐츠보드 미설정 점검 → CTR 회복 → 노출 순위 상승"
    section: 광고 전략 (CPC)
    priority: high

- id: package-never-enabled
  when: shop.package_orders == 0 and shop.package_enabled == false
  then:
    suggestion: "포장 주문 활성화로 수수료 절감 채널 확보"
    section: 배민 기본 세팅
    priority: medium
```

- 담당자가 룰 YAML만 추가하면 솔루션 계획서에 자동 반영되는 구조
- Claude API는 매칭된 룰들을 **자연스러운 문장으로 엮는 역할만** 담당
- 룰 자체는 코드가 아니므로 담당자가 직접 수정·추가 가능
