# 코딩 베스트 프랙티스

> 코딩 작업 시 적용되는 규칙

## API / 인터페이스

- 안정적인 인터페이스 기준으로 경계를 설계해라.
- 코드 경로 복제보다 선택적 매개변수 추가를 선호.
- 에러 의미론을 일관되게 유지 (throw vs return error vs 빈 결과).

## 테스트 전략

- 버그를 잡았을 최소한의 테스트를 추가해라.
- 순수 로직 → 유닛 테스트, DB/네트워크 경계 → 통합 테스트, 핵심 사용자 흐름만 → E2E.
- 구현 세부사항에 묶인 취약한 테스트를 피해라.

## 타입 안전성

- `any`나 억제(ignore) 사용은 프로젝트가 명시적으로 허용하고 대안이 없을 때만.
- 불변량은 경계에서 검증해라, 분산 검사가 아닌.

## 의존성 관리

- 기존 스택으로 깔끔하게 해결 가능하면 새 의존성을 추가하지 마라.
- 표준 라이브러리 / 기존 유틸리티를 선호.

## 보안

- 시크릿을 코드, 로그, 채팅 출력에 절대 넣지 마라.
- 사용자 입력은 신뢰하지 마라: 검증, 살균, 제한.
- 최소 권한 원칙 (특히 DB 접근과 서버 사이드 액션).

## 성능

- 성급한 최적화를 피해라.
- N+1, 무한 루프, 반복 무거운 연산은 바로 수정.
- 의심되면 추측하지 말고 측정해라.

## 접근성/UX (UI 변경 시)

- 키보드 내비게이션, 포커스 관리, 읽기 좋은 대비, 의미 있는 빈 상태/에러 상태.
- 화려한 효과보다 명확한 텍스트와 예측 가능한 인터랙션.
