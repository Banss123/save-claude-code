# Step 1: Think — 가설 검증 (Forcing Questions)

업무 시작 전, 6가지 강제 질문으로 가정을 두들겨본다. "이걸 진행하는 게 진짜 맞는가?"를 점검하는 단계.

## 6 Forcing Questions

1. **진짜 풀고 있는 문제는?** (사용자 × 상황 × 결과)
2. **가장 위험한 잘못된 가정은?** (이게 틀리면 모든 게 무너짐)
3. **성공 측정 기준은?** (정량/정성, 한 줄로)
4. **가장 작은 검증 가능 단위는?** (풀스코프 말고 슬라이스)
5. **안 만들면 뭐가 손해? (Null Hypothesis)** (손해 없으면 만들지 마)
6. **의사결정자는 누구?** (본인? 클라이언트? 데이터?)

## 비코딩 응용 예시

### 콘텐츠 작업 (3콘텐츠, 8앰비언트사운드)
- Q1: "구독자가 영상을 끝까지 안 봐서 알고리즘이 안 밀어줌"
- Q2: "썸네일 문제일 거야" → 사실은 후킹이 약한 것일 수 있음
- Q3: 시청 유지율 +10% / 댓글 5개 이상
- Q4: 1개 영상 A/B 테스트
- Q5: 손해 없으면 → 만들지 마. 명확한 손해 있어야 진행
- Q6: 시청 데이터 / 본인

### 영업/포워딩 (7어시스턴트)
- Q1: "클라이언트가 결정 못하고 있어"
- Q2: "정보 부족일 거야" → 사실은 의사결정자 부재일 수 있음
- Q3: 다음 미팅 잡힘 / 가격 합의
- Q4: 다음 메시지 1통으로 의사결정자 식별
- Q5: 손해: 매주 재커뮤니케이션 비용
- Q6: 클라이언트 측 결정권자

### 자료 정리/아카이브 (6아카이브)
- Q1: "나중에 이 자료를 못 찾겠다"
- Q2: "분류 체계가 문제" → 사실은 검색 키워드가 부재
- Q3: 6개월 뒤 검색 1회로 도달 가능
- Q4: 1개 자료로 태그 체계 시범 적용
- Q5: 손해: 매번 재검색 시간
- Q6: 미래의 나

## 산출물
- 6개 답변 (각 1~2줄)
- "진행" / "보류" / "재정의" 결정 (한 줄)

## 생략 조건
- 단축어 "그냥 해" → Step 1 (Think) + Step 2 (intake) 생략, Step 3부터 시작
- 작업이 명백한 후속 (이전 대화 이어가기) → 간단 확인만
- 단순 정보 조회 / 파일 이동 / 포맷 변환 → 생략 가능
- 사용자가 forcing question에 이미 답한 맥락이면 → 생략 가능
