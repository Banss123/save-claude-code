# 에러 처리 및 복구

## "라인 멈춤" 규칙
예상치 못한 일 발생 시 (테스트 실패, 빌드 에러, 동작 리그레션, 결과물 기대 불일치):

- 기능 추가를 멈춰라
- 증거를 보존해라 (에러 출력, 재현 단계, 산출물 원본)
- 진단으로 돌아가서 재계획해라

## 트리아지 순서
1. 확실하게 재현
2. 실패 지점 격리 (UI, API, DB, 네트워크, 빌드 도구, 산출물 중 어디?)
3. 최소 실패 케이스로 축소
4. 근본 원인 수정 (증상이 아닌)
5. 회귀 테스트 추가 (코딩) / 체크리스트 추가 (비코딩)
6. 원래 보고 기준으로 엔드투엔드 검증

## 안전한 폴백
- 부분 동작보다 "안전한 기본값 + 경고" 선호
- 조용한 실패가 아닌 실행 가능한 에러 반환
- 수정으로서의 광범위 리팩토링은 피해라

## 방향이 틀렸을 때
- 즉시 멈추고 현재까지 산출물 보존
- 어디서부터 틀어졌는지 지점 특정 → 대안 2~3개 제시
- 살릴 수 있는 중간 산출물 분류
