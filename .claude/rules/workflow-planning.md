# Step 4: 실행 계획 수립

- 업무를 단계별로 쪼개서 실행 계획을 제시한다.
- 각 단계의 목적과 예상 결과물을 한 줄로 적는다.
- 복잡한 업무는 에이전트 팀 구성을 제안한다:
  - 어떤 서브에이전트가 필요한지
  - 각 서브에이전트에 적합한 모델 배정 (메인 CLAUDE.md 모델 라우팅 참고)
- 계획에 **검증 단계를 반드시 포함**한다 (나중에 추가하지 말고 처음부터).
- 계획을 사용자가 승인하면 Step 5로 진행한다.

---

## 4-way 다중 리뷰 (계획의 압박 테스트)

복잡하거나 영향이 큰 작업은 계획안을 4명의 가상 페르소나로 압박 테스트한다.
gstack의 `/plan-*-review` 패턴을 사용자 환경에 맞춘 형태.

### 페르소나 정의 우선순위

1. **프로젝트 CLAUDE.md에 정의된 4-way 페르소나** → 있으면 그대로 사용
2. 없으면 → **메인 CLAUDE.md의 글로벌 디폴트 4명** (CEO / 엔지니어 / 디자이너 / DevEx)

### 라우팅 지능 (gstack 컨셉 그대로)

모든 4명을 다 부르는 게 아니라 **작업 성격에 따라 1~3명만 호출**:

| 작업 성격 | 호출 페르소나 수 | 예시 |
|---|---|---|
| 단순 자료 정리 / 포맷 변환 | 1명 | 검색가능성 검토자 |
| 콘텐츠 톤 잡기 / 카피 다듬기 | 2명 | 시청자대표 + 카피라이터 |
| 자동화 설계 / 시스템 변경 | 3명 | 자동화엔지니어 + 보안 + DX |
| 큰 전략 결정 / 신규 프로젝트 | 4명 다 | 전 페르소나 |

### 호출 형식 (각 페르소나)

- **가장 큰 우려 1개** (구체적으로)
- **보강 제안 1개** (실행 가능하게)
- **평결**: 통과 / 보류 / 반대

### 산출물

- 페르소나별 짧은 코멘트 (3~5줄)
- **통합 평결**: 계획 확정 / 수정 필요 / 재설계
- 사용자에게 보여주고 최종 승인

### 4-way 리뷰 생략 조건

- 작업이 단순하고 영향 작음
- 사용자가 "그냥 해" / "전부 맡겨" 단축어 사용
- 같은 패턴 반복 작업 (이전에 4-way 통과)
